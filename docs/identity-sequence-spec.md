# DaisyBase Identity And Sequence Specification

## 1. Objective

Add engine-native sequence objects and identity-backed columns so DaisyBase can generate durable numeric keys inside the engine and expose them through JDBC `RETURN_GENERATED_KEYS` without client-side emulation.

This work must fit the existing architecture:

- immutable catalog snapshots
- durable catalog persistence
- live heap storage with WAL-backed recovery
- custom SQL frontend and JDBC driver

## 2. Scope

### 2.1 In Scope

- `CREATE SEQUENCE`, metadata persistence, restart durability, and transactional DDL visibility
- `NEXT VALUE FOR sequence_name` expression support
- table columns declared as identity-backed numeric columns
- implicit value generation on `INSERT`
- JDBC generated-key result sets for single-row and batched inserts when keys are engine-generated
- remote protocol compatibility through the existing statement execution path

### 2.2 Out Of Scope For This Slice

- sequence caching across sessions
- descending sequences, cycling, min/max bounds beyond basic defaults
- `ALTER SEQUENCE`
- identity `SET GENERATED` variants beyond initial `BY DEFAULT` / `ALWAYS`
- SQL-standard `DEFAULT next value for ...` rewrite for arbitrary expressions unless it is required internally

## 3. SQL Surface

### 3.1 Sequence DDL

Support:

- `CREATE SEQUENCE seq_name;`
- `CREATE SEQUENCE schema.seq_name START WITH 1 INCREMENT BY 1;`

Initial defaults:

- `START WITH 1`
- `INCREMENT BY 1`
- `NO CYCLE`
- integer domain is `BIGINT`

Unsupported clauses must fail with `UNSUPPORTED_FEATURE`.

### 3.2 Identity Column DDL

Support on `INTEGER` and `BIGINT` columns only:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY`
- `id BIGINT GENERATED ALWAYS AS IDENTITY`
- optional `PRIMARY KEY` / `UNIQUE` / `NOT NULL`

Compatibility rule:

- identity implies `NOT NULL`
- `GENERATED ALWAYS` rejects explicit inserted values unless an explicit override syntax is later added
- `GENERATED BY DEFAULT` accepts explicit values and only generates when the column is omitted or `NULL`

### 3.3 DML Semantics

- omitted identity column value generates from the backing sequence
- `NULL` on `BY DEFAULT` identity generates a value
- `NULL` on `ALWAYS` identity generates a value
- explicit value on `ALWAYS` identity fails with `CONSTRAINT_VIOLATION`
- `NEXT VALUE FOR seq` is evaluated once per row
- sequence advancement is never rolled back after consumption

## 4. Catalog Model

Extend catalog definitions with durable sequence and identity metadata.

### 4.1 New Catalog Objects

Add:

- `SequenceDefinition`
  - object id
  - qualified name
  - data type: `BIGINT`
  - start value
  - increment
  - next value
- identity metadata on `ColumnDefinition`
  - generation kind: none / by-default / always
  - optional backing sequence object id

### 4.2 Snapshot Rules

- sequence definitions are visible through immutable `CatalogSnapshot`
- sequence current value is operational state and must not be treated as a compile-time constant
- DDL that creates a sequence or identity-backed table becomes visible only after commit

### 4.3 Persistence Format

Update catalog snapshot serialization and change-log serialization to include:

- `CREATE_SEQUENCE`
- column identity metadata in table definitions

Do not break existing catalogs silently. Persisted format version must be bumped or backward-compatible parsing added.

## 5. Storage And Recovery

### 5.1 Sequence State Storage

Persist sequence state in a dedicated durable structure under the database home. Acceptable initial implementations:

- dedicated sequence-state file
- catalog-adjacent state file
- heap-backed system table if done cleanly

Requirements:

- state survives restart
- state flush is WAL-protected or ordered with equivalent durability guarantees
- advancing a sequence must not require rewriting the whole catalog snapshot

### 5.2 WAL Requirements

Add WAL records for sequence advancement and identity-generated insert metadata, or ensure page-image logging captures the durable state change with correct ordering.

Required semantics:

- committed inserts that consumed generated keys must recover with the same persisted inserted values
- post-crash sequence `nextValue` must be greater than any committed generated identity value already stored
- gaps are acceptable after rollback or crash; reuse is not

## 6. SQL Frontend

### 6.1 Parser

Add AST support for:

- `CreateSequenceStatement`
- `NextValueExpression`
- column identity clause in `CREATE TABLE`

Parsing rules must preserve source spans and fail explicitly on unsupported sequence options.

### 6.2 Binder

- bind `NEXT VALUE FOR seq_name` against catalog snapshots
- reject unknown sequences with `SEMANTIC_ERROR`
- reject identity clauses on non-numeric columns
- record identity metadata in bound table definitions / insert plans

## 7. Planner And Execution

### 7.1 Insert Planning

Insert plans must know:

- which columns are omitted
- which omitted columns require identity generation
- which generated values need to be surfaced back to JDBC as generated keys

### 7.2 Runtime Generation

At execution time:

- allocate sequence values inside the engine, not in JDBC
- apply generated values before constraint checks and index maintenance
- materialize generated keys in statement results for inserts that request them

### 7.3 JDBC Contract

- `Statement.getGeneratedKeys()` and `PreparedStatement.getGeneratedKeys()` must return actual generated identity values for inserts that generated them
- for batched inserts, generated keys must be ordered to match successful insert execution order
- if no keys were generated, return an empty result set

## 8. Engine API And Protocol

No separate protocol message is required if generated keys travel in normal statement results.

Implementation options:

- extend `StatementResult` with generated-key batches
- or extend `commandTag`/batch conventions in a backward-compatible way

Preferred direction:

- add explicit generated-key payload to `EngineApi.StatementResult`
- extend remote protocol serialization accordingly
- update JDBC result materialization to consume that payload instead of faking an empty result set

## 9. Testing Requirements

### 9.1 Unit Tests

- parser coverage for sequences and identity clauses
- catalog serialization round-trip for sequence definitions and identity metadata
- sequence-state persistence and restart recovery
- binder rejection for invalid identity declarations and unknown sequences

### 9.2 Integration Tests

- create sequence, consume values, restart, consume more values
- identity insert with omitted column returns generated keys
- `GENERATED ALWAYS` rejects explicit values
- `GENERATED BY DEFAULT` accepts explicit values and resumes generation afterwards
- uniqueness and primary-key enforcement still holds for generated keys
- remote JDBC generated keys match embedded behavior

### 9.3 Stress Tests

- concurrent inserts into one identity-backed table produce distinct keys
- rollback creates gaps but never duplicates
- crash between WAL flush and page flush preserves committed generated keys

## 10. Acceptance Criteria

This slice is done only when:

- generated values come from the engine
- generated keys are durable and survive restart
- JDBC returns real generated keys
- sequence/identity behavior is identical in embedded and remote modes
- full verification passes:
  - `./gradlew.bat :sql-frontend:test :catalog:test :storage:test :engine-api:test :jdbc:test`
  - `./gradlew.bat build`

## 11. Recommended Ownership

Primary modules:

- `sql-frontend`
- `catalog`
- `storage`
- `engine-api`
- `jdbc`

Tests to add or expand:

- parser tests
- catalog persistence tests
- engine integration tests
- JDBC embedded and remote tests
